# 01_PRD — 환자 콘텐츠 링크/QR 접근 보안 (옵션 C: 민감도 분리)

> 대상: `clinic-content-system` 산출물(진료설명 decks · 유인물 handouts · 검사결과 lab-reports)의 공유 링크/QR 접근 통제.
> 작성: 2026-06-12 (show-me-the-prd) · SoT=이 repo · **승인 방향: 옵션 C(민감도 분리), 검진결과 확장 대비**

## 1. 한 줄
**개인정보 없는 일반 교육자료는 가볍게(공개+noindex), 환자 식별정보가 든 검사결과는 진짜로 잠근다** — 만료되는 서명 링크 + (선택) 이름·환자번호 로그인 + 추측불가 URL. 향후 **검진결과 페이지도 같은 보호 티어에 끼워넣게** 설계.

## 2. 문제 정의
- 현재 모든 산출물이 GitHub Pages(정적)에 **공개**(noindex만). 검사결과(lab-reports)는 **환자명+차트번호 PII**를 담는데, 추측불가 해시 URL·QR제거뿐 접근통제는 없음.
- 정적 호스팅은 서버가 없어 '시간 지나면 자동으로 닫힘'·로그인·만료를 **자체적으로 못 함**.
- 카톡/문자로 링크·QR을 보내면 전달·캡쳐로 **무기한·무제한 확산** 가능.

## 3. 목표 & 비목표
**목표**
- G1. 검사결과(PII) 링크는 **만료**된다(예: 발급 후 24h, 조정가능). 만료 후 접근 차단.
- G2. 검사결과는 **이름+환자번호** 확인(또는 서명 링크 단독)으로만 열림 — "기본 보안".
- G3. 일반 교육자료(진료설명·유인물, PII 없음)는 마찰 없이 공개 유지(noindex).
- G4. **검진결과(향후 신설) 페이지**를 코드 변경 최소로 검사결과와 **같은 보호 티어**에 추가 가능.
- G5. 원장이 링크 발급/만료/공유를 **간단히** 할 수 있다(진료 흐름 방해 0).

**비목표**
- 병원정보시스템(EMR)급 인증·감사, 환자 계정 시스템, 법적 의무기록 보관 — 이번 아님.
- 일반 교육자료의 강한 인증(과함).

## 4. 사용자 & 위협 모델
- **원장**: 진료 중 검사결과 링크/QR을 환자에게 공유. 발급·공유가 빨라야 함.
- **환자**: 링크/QR을 받아 본인 결과 열람(이름+환자번호 또는 유효기간 내 링크).
- **위협**: ①링크 전달/유출로 타인 열람 ②QR 캡쳐 후 장기 노출 ③URL 추측 ④검색엔진 노출.
- **현 방어**: 해시 URL(추측 방지)·noindex. **부족**: 만료·인증·유출 후 무력화.

## 5. 접근 방식 — 옵션 C (민감도 2티어)
| 티어 | 대상 | 호스팅 | 보호 |
|------|------|--------|------|
| **T0 공개** | 진료설명 decks · 유인물 handouts (PII 없음) | GitHub Pages 유지 | 공개 + noindex (현행) |
| **T1 보호** | **검사결과 lab-reports** + **(향후) 검진결과** | **Vercel(엣지 미들웨어)** | 만료 서명링크 + 이름·환자번호 게이트 + 해시 URL + noindex/no-store |

**왜 Vercel**: 이미 트레이딩 대시보드로 Vercel 사용 중(Basic Auth 엣지 인증 경험). 엣지 미들웨어로 정적자산 앞단에서 토큰 검증·만료·차단이 정적 호스팅에선 불가능한 걸 가능케 함.

## 6. 요구사항
### 기능(FR)
- **FR1 만료 서명 링크**: 보호 자료 URL에 HMAC 서명 토큰(만료시각 포함). 엣지가 서명·만료 검증, 실패 시 차단(만료/무효 안내 페이지).
- **FR2 발급 도구**: 원장이 자료에 대해 "유효기간 N시간" 링크/QR을 한 번에 생성(CLI 또는 스킬 명령). 기본 만료값 설정.
- **FR3 이름+환자번호 게이트(선택/2차)**: 보호 페이지 진입 시 이름+환자번호 입력 → 엣지가 해당 자료의 기대값과 대조. 저엔트로피라 **반드시 만료링크·해시URL과 결합** + 시도 제한(rate-limit).
- **FR4 티어 라우팅(확장성)**: 보호 대상 `kind` 목록(config)으로 `lab-reports`·`checkup-results`(향후) 지정 → 같은 엣지 게이트 자동 적용. 새 종류 추가 = config 한 줄.
- **FR5 헤더**: 보호 페이지 `no-store`·`noindex`·referrer-none, 캐시·검색 노출 차단.
- **FR6 무력화**: 유출 의심 시 링크 즉시 만료(서명 키 회전 또는 자료별 revoke 리스트).

### 비기능(NFR)
- NFR1 **간단·저마찰**: 환자는 링크 클릭(+이름/번호) 한 번. 원장은 발급 1스텝.
- NFR2 **기존 무손상**: 일반 교육자료·노션 라우팅·빌드 파이프라인 유지. 보호 티어만 신규.
- NFR3 **PII 최소**: 페이지/로그에 환자번호 평문 저장·노출 금지(해시 비교), 이름·번호는 검증에만.
- NFR4 **비용 0~저**: Vercel 무료/저비용 범위.

## 7. 성공 지표
- M1 검사결과 링크가 만료 후 100% 차단.
- M2 URL 추측·검색 노출 0(해시+noindex+no-store).
- M3 원장 발급 1스텝, 환자 열람 마찰 최소(클릭+번호).
- M4 검진결과 페이지 신설 시 보호 적용이 config 한 줄로 됨.

## 8. 리스크 & 가정
- R1 환자번호 저엔트로피 → 단독 인증 불가. **만료링크+해시URL 필수 결합**으로 완화.
- R2 Vercel 이전 시 URL/QR 도메인 변경 → 보호 티어만 이전, 기존 발급물 마이그레이션 안내.
- R3 엣지 미들웨어 오설정 = 공개 노출 사고 → fail-closed(검증 실패=차단) + 스모크 테스트.
- 가정: 옵션 C 승인됨. 검진결과는 향후 신설이며 이번 설계는 그 자리(config 슬롯)만 마련.

## 9. 미해결 질문
- 1차 인증을 '만료 서명링크 단독'으로 갈지 '+이름·환자번호'까지 둘지(환자 편의 vs 보안) 기본값.
- 기본 만료시간(24h? 진료당일만?).
- 보호 티어를 Vercel 별도 프로젝트로 둘지, 기존 대시보드 프로젝트에 경로로 합칠지.
