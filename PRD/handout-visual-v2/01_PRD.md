# 01_PRD — 환자설명 핸드아웃 스킬 v2: 비주얼·레이아웃·톤 하드닝

> 대상 스킬: `clinic-content-system`의 **handouts** 파이프라인 (광교바른내과 진료용)
> 작성: 2026-06-12 · 인터뷰 기반(show-me-the-prd) · SoT = 이 repo

## 1. 한 줄 요약
환자가 **글이 아니라 그림으로 내용을 이해**하도록, 핸드아웃의 ①이미지 관련성(겉도는 이미지 제거) ②이미지 생성 안정성 ③레이아웃 깨짐 0 ④톤·디자인 일관성을 **자동 게이트로 강제**하는 v2 업데이트.

## 2. 문제 정의 (사용자 페인포인트 — 2026-06-12)
| # | 현상 | 현재 스킬의 한계 |
|---|------|-----------------|
| P1 | **겉도는 이미지** — 이미지가 전체 내용을 아우르지 못하고 분위기만 더함 | §15에 "generic 금지" 룰은 있으나 *생성된 이미지가 실제 본문을 설명하는지* 검증하는 자동 게이트가 없음 → 주관적 판단에 의존 |
| P2 | **이미지 생성 불안정성** | `$imagegen`/`codex_imagen.sh` 경로가 driver·환경 의존, 실패 시 생략으로 끝남. 재시도·후보 비교·품질 QA 루프 부재 |
| P3 | **레이아웃 깨짐** | `_validate_layout`(bbox)+`_visual_audit`가 있지만 "검증기 통과 ≠ 시각 통과"(SKILL.md 명시). 이미지 삽입 후 회귀(겹침·잘림·여백)가 수동 육안에 의존 |
| P4 | **톤·디자인 비일관** | `brand-design-system.md` 토큰은 있으나 자료 간 톤·이미지 스타일·색·타이포 일관성을 **측정·강제**하는 장치가 없음 |

## 3. 목표 & 비목표
**목표(이번 v2)**
- G1. 모든 핸드아웃 이미지가 **본문 핵심 1개를 명시적으로 설명**(aboutness 게이트 통과)하지 못하면 채택 안 됨.
- G2. 이미지 생성 **재시도·후보 N장 비교·자동 QA**로 "불안정→생략"을 "안정→채택" 으로.
- G3. 이미지 삽입 후 **레이아웃 회귀 자동 차단**(삽입 전/후 visual diff + 기존 검증기 강화).
- G4. **브랜드 톤 일관성 스코어**(색·타이포·이미지 스타일)를 빌드 게이트에 추가.

**비목표(이번 아님)**
- 주제 라이브러리 대량 확충(별도 PRD), 환자 개인화 변수 주입, 다국어, 모바일/카톡 워크플로 — 모두 후속.

## 4. 사용자 & 사용 맥락
- **주 사용자**: 원장(정지환) — 진료/진료 외 시간에 "OO 핸드아웃 만들어줘" 호출. 1인 운영, 저빈도·고품질 요구.
- **최종 수신자**: 광교바른내과 환자(특히 노년층) — 글보다 그림으로 이해. 인쇄물(A4) + 카톡 공유 라이브 URL.
- **실행 주체**: Claude/Codex 에이전트(스킬). 사람 개입 최소(자동 생성·검증·커밋).

## 5. 요구사항
### 5.1 기능 요구 (FR)
- **FR1 이미지 aboutness 게이트**: 슬롯마다 "이 이미지가 설명할 본문 문장/표 1개"를 명시(intent) → 생성 후 **VLM 교차검증**(이미지가 그 intent를 실제로 묘사하는가 Y/N + 점수). 미달이면 재생성 또는 슬롯 제거.
- **FR2 안정 생성 루프**: imagegen 호출에 재시도(≥2)·후보 2~3장 생성→aboutness/품질 최고 1장 채택. 경로 실패 시 fallback 체인(codex_imagen → $imagegen → 명시적 생략+사유 기록).
- **FR3 레이아웃 회귀 가드**: 이미지 삽입 **전/후 스냅샷 비교**(visual diff) + `_validate_layout` 강화 체크가 모두 통과해야 채택. 깨지면 슬롯 크기/배치 자동 보정 후 재검.
- **FR4 톤 일관성 스코어**: 빌드 시 색 팔레트·폰트·이미지 스타일을 brand 토큰과 대조해 일관성 점수 산출, 임계 미달 경고/차단.
- **FR5 보고**: 완료 시 "이미지 N장(각 intent·aboutness 점수), 레이아웃 0깨짐, 톤 점수"를 사용자에게 요약.

### 5.2 비기능 요구 (NFR)
- NFR1 **결정론·재현성**: 같은 입력→같은 슬롯/검증 결과. 이미지 자산은 버전관리·재사용.
- NFR2 **사람 개입 0** 기본(자동 생성·검증·커밋·푸시). 실패만 사용자에게 에스컬레이션.
- NFR3 **하위호환**: 기존 decks/lab-reports 파이프라인·Notion 라우팅 무손상. handouts에 먼저 적용 후 확장.
- NFR4 **멀티머신 전파**: 룰=repo 커밋(push), 사용자 선호만 메모리.

## 6. 성공 지표
- M1 채택 이미지의 aboutness 게이트 통과율 100%(미달은 생략으로 분류, "겉도는 이미지" 0건).
- M2 이미지 생성 성공률(재시도 포함) ↑, "불안정→생략" 비율 ↓.
- M3 푸시된 핸드아웃의 레이아웃 깨짐 0건(visual diff 회귀 0).
- M4 톤 일관성 스코어 ≥ 임계, 자료 간 시각 일관성 사용자 체감 개선.

## 7. 리스크 & 가정
- R1 VLM aboutness 판정의 신뢰도 — 룰/프롬프트 캘리브레이션 필요(초기 사람 스팟체크).
- R2 imagegen 환경 의존(codex CLI 로그인 등) — fallback·기록으로 완화.
- R3 visual diff 오탐(정상 변경을 회귀로) — 임계·마스킹 튜닝.
- 가정: $imagegen/codex_imagen 경로는 유지(이번엔 신뢰성·QA만 보강), 브랜드 토큰은 현 SoT 유지.
