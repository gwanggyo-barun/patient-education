# 덱 레이아웃 디자인 비율 룰 (황금비율 / 모듈러 스케일)

> 2026-06-06 제정. 사용자 검수 + 코덱스(시니어 디자이너 관점) 감수 반영. 대상: clinic-content-system **deck (1280×720)**. 목적: "폰트 적당(작지도 크지도 않게)·여백 대칭·박스와 이미지 안 붙음"을 수치로 보장. 근거: A List Apart 모듈러 스케일, Figma 황금비율, Cieden type-scale(1.618 고대비/1.333 고밀도).

## 1. 타입 스케일 (base 16px=1rem, Perfect Fourth 1.333 + 타이틀 황금 강조)
노안 환자 포함 가독성 우선. **현재 visual-focus의 0.58~0.78rem 강제 축소는 전면 폐기.**

| 역할 | px | rem | 줄간격 |
|---|---:|---:|---:|
| 슬라이드 제목 | 36~40 (강조형만 42) | 2.25~2.5 | 1.12~1.2 |
| 부제 | 19~21 | 1.19~1.31 | 1.3~1.4 |
| 박스 제목 | 20~22 | 1.25~1.375 | 1.2~1.3 |
| 박스 라벨/강조 | 17~18 | 1.06~1.125 | 1.35~1.45 |
| 박스 본문 | **16~17 (하한 15.5)** | 1.0~1.0625 | 1.45~1.55 |
| 표 셀 | **15~16 (하한 15)** | 0.94~1.0 | 1.35~1.45 |
| 표 헤더 | 15~16 | 0.94~1.0 | 1.25~1.35 |
| 표 보조 텍스트 | 15 | 0.9375 | 1.35~1.45 |
| 출처/캡션 | 13~14 (출처만 12.5~13) | 0.81~0.875 | 1.25~1.35 |

한글 본문 좌측정렬(양쪽정렬 금지). 한 줄 18~34자 권장(콘텐츠 열 너무 넓으면 max-width).

## 2. 박스 내부 여백·정렬 — 언더필(비대칭 여백) 해결
근본 원인 = 박스가 이미지 높이에 억지로 stretch → 내용 위 몰림 + 아래 큰 공백. 우선순위:
1. **기본: 박스 높이를 내용에 맞춤**(과대 stretch 금지). 카드 스택 전체 높이를 이미지와 균형.
2. **이미지 높이 = 좌측 콘텐츠 스택의 0.95~1.08배** 안에서. (이미지만 과하게 크면 불균형)
3. **부득이 stretch 시에만 `justify-content: center`**(세로 중앙) → 위아래 여백 대칭. 일반 정보 카드는 상단정렬 + 높이를 내용에 맞춤.
4. 같은 행 카드끼리만 등높이(stretch). 좌측 세로 스택 전체를 이미지에 stretch 금지.
- 박스 패딩: 세로 ≈ 본문폰트×1.0~1.3(16px→16~20px), 가로 = 세로×1.333(22~28px).
- 카드 radius 8~12px 전 덱 통일. border 1px 중심, 그림자는 강조 카드만 약하게.
- 카드 gap: 스택 12~16px, 그리드 14~18px (콘텐츠-이미지 거터보다 작게 = 위계).
- 텍스트 좌측 시작 일치, 숫자 표만 우측정렬([[Gotcha 17]]). 표는 zebra 또는 divider 중 하나만.

## 3. 본문↔이미지 분할·거터 (콘텐츠 의존, 1:1.618 고정 폐기)
| 상황 | 콘텐츠:이미지 |
|---|---:|
| 이미지 주도 + 카드 2~3개 | 0.62 : 1 |
| 일반 visual-focus, 카드 3~4개 | 0.72 : 1 (≈5:7) |
| 콘텐츠+이미지 균형 | 1 : 1 ~ 6:5 |
| 표/카드 5~6개 | 콘텐츠 최소 520px 확보, 이미지 축소 |
| 표 중심 | 콘텐츠 1.2~1.6, 이미지 보조 |
- **콘텐츠-이미지 거터: 권장 32px, 최소 28px, 차단 24px.** (박스가 이미지에 붙어 '겹쳐 보임' 방지)
- 이미지 radius ≤ 카드, 정보전달용은 어두운 overlay·과한 blur·장식 그림자 금지. 캡션은 이미지 안쪽 하단 한 위치 통일(텍스트 박스가 이미지 영역 침범 금지).

## 4. 슬라이드 채움 균형 (꽉도 비지도 않게) — 2026-06-06 코덱스 2차 감수 보강
- 본문 최하단~푸터 상단 여백: **24~56px 적정**. `<16px` 과밀, `>72px` 언더필.
- **박스 내용 넘침 금지**: `scrollHeight > clientHeight+2`면 차단(box_content_overflow). 긴 비교값("20.2% vs 13.7%")은 폰트 축소보다 **라벨/값/증감 분리** 권장. stat 값 1.45~1.6rem(최대 1.7), line-height 1.05~1.15, 카드 내부 상하 여유 10~14px.
- **핵심 원칙**: "대칭이면 OK"가 아니라 **박스 크기가 콘텐츠 무게와 맞아야 한다.** 콘텐츠 적은데 박스만 크면 병원자료 완성도 저하.
- **희소 슬라이드 자동 판정** (`inner_fill = 보이는 내용높이 / 박스 내부높이`):
  | inner_fill | 처리 |
  |---|---|
  | ≥ 0.55 | A: 그대로 OK |
  | 0.40~0.55 | B: 콘텐츠 확대(폰트 아님 — 패딩·행간·막대높이·아이콘) |
  | 0.28~0.40 | C: 박스를 내용 높이로 줄이고 그룹 중앙/상단 45% 배치 |
  | < 0.28 | D: 재설계(카드 추가·해석카드·이미지) |
  - 기본 순서 **B→C→D**, A는 inner_fill ≥0.55 또는 의도적 조용한 슬라이드(closing·quote)만. `body_fill_ratio < 0.62`면 D 우선 검토.
- **표·리스트 채움**: 행 ≥4 → `space-between`(행간 상한 표 18~22px·리스트 20~24·카드 24~28); 행 2~3 → `space-between 금지`, 박스 높이 = 내용+48~72px. 행간이 상한 넘으면 박스 축소 또는 내용 보강.
- **막대그래프**: bar 두께 14~20px, row 높이 44~56px, row gap 14~20px, 채움 0.50~0.70. <0.40 → B로 키움, <0.45 → C로 패널 축소, 데이터 ≤3 → D(해석카드·기준선·전후비교).
- **카드 2개 구성**: 각 높이 상한 220~260px, **full-height stretch 금지**(안 그럼 350px 텅 빔). 내부 채움 목표 0.50~0.68. <0.40 → C, <0.30 → D(3카드화·요약 strip 추가).

## 5. 컬러·접근성
- 본문 WCAG AA 대비 ≥ 4.5:1. Steel Blue(#5B9BD5) 위 흰 텍스트는 크기·굵기 확인(작은 본문은 위험 → navy 사용).

## 6. 검증기 자동 탐지 (`shared/_validate_layout.py` DECK)
- `body_overlaps_footer`, `body_underfills`(보이는 박스 최하단 기준 `>72px`), `box_underfill`(하단공백 비대칭 `>28px AND 상단보다 >24px`), `box_content_overflow`(음수 slack=실제 넘침), `sibling_box_overlap`(형제 카드 물리 겹침), `font_too_small`(본문 `<15px` 차단, kicker/label/value 면제), `content_image_gutter`(`<24px`), `sparse_box`(대칭이지만 inner_fill < 0.48 — 경고).
- ⭐⭐ **검증·캡처는 PDF와 동일한 `print` 미디어로 (2026-06-06 슬라이드2 겹침 사고).** PDF는 `emulate_media('print')`로 렌더되는데, screen 미디어 검증은 print 의 미세 폰트 메트릭 차이로 생기는 카드 겹침(DOM 1px 여유 → PDF 2~3px 넘침)을 놓친다. `_validate_layout`·build·`slide_screens` 모두 deck 은 print 모드 + 폰트 로딩 1.2s 대기.
- ⭐ **squeeze된 박스(고정/grid 높이로 콘텐츠가 꽉 끼는 경우)는 min-height 로 양의 여유(4px+) 확보.** stat-grid 3카드+note 같은 빡빡한 칸 필수.
- ⭐ **ground-truth = 실제 PDF 픽셀.** 의심되면 PyMuPDF(`fitz`)로 PDF 페이지를 3x 렌더해 카드 rect 겹침을 직접 확인 (DOM getBoundingClientRect ≠ PDF 래스터라이저).
- ⭐ **Step 3.8: PDF가 아니라 슬라이드별 PNG를 한 장씩 전부 눈으로 확인. 검증기 통과 ≠ 시각 통과.** 체크: 박스 대칭·내용 넘침·카드 겹침·폰트 스케일·거터·표 정렬·희소박스.

## 적용
visual-focus 강제 축소 블록을 위 스케일로 교체. 신규 덱은 처음부터 이 룰로. 예외 슬라이드만 deck-local override.
